## 1. エグゼクティブサマリー

### 1.1 目的

Context7 MCPの機能を分析し、YATAがそれを上回る機能を提供するための要件を定義する。

### 1.2 Context7の主な機能

| 機能 | 説明 |
|------|------|
| `resolve-library-id` | ライブラリ名からContext7互換IDを解決 |
| `query-docs` | ライブラリIDを使用してドキュメントを取得 |
| バージョン指定 | 特定バージョンのドキュメントを取得 |
| 外部ライブラリドキュメント | クラウドからリアルタイムでドキュメント取得 |

### 1.3 Context7の限界

| 限界 | 説明 |
|------|------|
| ローカルコード非対応 | ユーザーのプロジェクトコードを解析しない |
| 外部依存 | クラウドAPIに依存（オフライン不可） |
| コード関係性非解析 | 関数間の呼び出し関係を追跡しない |
| プライベートコード非対応 | 社内ライブラリのドキュメントなし |
| 静的ドキュメントのみ | コード実行結果や型情報なし |

### 1.4 差別化戦略

YATAは以下の軸でContext7と**補完的に差別化**する:

| 軸 | 説明 |
|----|------|
| **ローカルファースト** | プライベートコードの完全理解、社内ライブラリ対応 |
| **関係性重視** | 単なるドキュメントでなく、コードの繋がり（calls, imports, inherits）を提供 |
| **オフライン対応** | ネットワーク不要で動作、セキュアな環境でも利用可能 |
| **統合分析** | ローカルコード + フレームワーク知識の融合検索 |
| **インパクト分析** | 変更影響範囲の可視化（Context7にない機能） |

**Context7との共存モデル**:
```
┌─────────────────────────────────────────────────────────┐
│                    AI Coding Assistant                   │
├─────────────────────────┬───────────────────────────────┤
│       Context7          │            YATA               │
│  ・外部ライブラリ最新API  │  ・ローカルコード理解          │
│  ・バージョン別ドキュメント│  ・プロジェクト固有パターン    │
│  ・クラウド依存           │  ・オフライン動作              │
│  ・汎用的なコード例       │  ・プロジェクト規約に沿った提案 │
└─────────────────────────┴───────────────────────────────┘
```

## 3. Context7を超える新機能要件

### 3.1 P0（Must Have）- 即座に価値を提供

#### REQ-001: インテリジェント・ドキュメント生成

**説明**: ローカルコードから自動的にドキュメントを生成し、Context7のようなドキュメント参照体験を提供

**ユースケース**:
- `generate_docs("UserService")` → クラスのドキュメント、使用例、関連コードを生成
- 既存コードのdocstringとコメントを解析してドキュメント化

**出力フォーマット**:
- `markdown`: 人間可読なMarkdown形式（デフォルト）
- `json`: 構造化データ形式（プログラム処理用）
- `html`: HTMLレンダリング用

**受け入れ基準**:
- [ ] エンティティのドキュメントをMarkdown/JSON/HTML形式で生成
- [ ] 使用例を自動抽出（最大10件、関連性スコア順）
- [ ] 関連するエンティティへのリンクを生成
- [ ] docstring、コメント、型ヒントを統合
- [ ] 日本語コメント/docstringに対応

#### REQ-002: コードスニペット推薦

**説明**: Context7の「コード例取得」を超え、ローカルコードベースから最適なコードスニペットを推薦

**ユースケース**:
- `recommend_code("authentication middleware")` → プロジェクト内の類似実装を提案
- フレームワーク知識グラフとローカルコードを組み合わせた提案

**パラメータ定義**:
- `query`: 検索クエリ（自然言語または技術用語）
- `include_frameworks`: フレームワーク知識を含むか（デフォルト: true）
- `include_local`: ローカルコードを含むか（デフォルト: true）
- `limit`: 最大結果数（デフォルト: 10、最大: 50）
- `min_relevance`: 最小関連性スコア（デフォルト: 0.3）

**関連性スコア算出**:
```
relevance_score = name_match * 0.4 + docstring_match * 0.3 + context_match * 0.2 + usage_frequency * 0.1
```

**受け入れ基準**:
- [ ] クエリに基づくコードスニペット検索（TF-IDF + セマンティック類似度）
- [ ] 関連性スコア（0.0〜1.0）によるランキング
- [ ] フレームワークとローカルコードの統合検索
- [ ] 結果にソース種別（local/framework）を明示
- [ ] コードスニペットの前後コンテキスト（±5行）を含む
- [ ] 使用頻度の高いスニペットを優先
- [ ] 結果が0件の場合、類似クエリを提案

**エラー処理**:
- 知識グラフ未構築: 「先にparse_directoryを実行してください」メッセージ
- クエリ空: デフォルトで最近使用されたエンティティを返却

#### REQ-003: 依存関係インパクト分析

**説明**: 変更の影響範囲を自動分析（Context7にない機能）

**ユースケース**:
- `analyze_impact("UserService.authenticate")` → この関数を変更した場合に影響を受けるコードを一覧表示
- リファクタリング前の影響調査

**パラメータ定義**:
- `depth`: 探索深度（デフォルト: 3、最小: 1、最大: 10）
- `include_indirect`: 間接依存を含むか（デフォルト: true）
- `min_impact_score`: 最小影響度スコア閾値（デフォルト: 0.1）

**影響度スコア算出式**:
```
impact_score = 1.0 / (depth ^ 1.5) * relationship_weight
relationship_weight: calls=1.0, imports=0.8, inherits=0.9, contains=0.5
```

**受け入れ基準**:
- [ ] 直接的な呼び出し元の特定（depth=1）
- [ ] 間接的な依存関係の追跡（depth=2〜10）
- [ ] 影響度スコアの算出（0.0〜1.0）
- [ ] 結果を影響度スコア降順でソート
- [ ] 循環依存の検出と警告

### 3.2 P1（Should Have）- 競争優位性

#### REQ-004: ハイブリッド検索（ローカル + フレームワーク）

**説明**: ローカルコードとフレームワーク知識を統合した検索

**ユースケース**:
- `hybrid_search("how to implement JWT authentication")` 
  → フレームワークのベストプラクティス + プロジェクト内の既存実装を組み合わせて提案

**パラメータ定義**:
- `query`: 検索クエリ
- `local_weight`: ローカル結果の重み（デフォルト: 0.5、範囲: 0.0〜1.0）
- `frameworks`: 検索対象フレームワーク（デフォルト: 自動検出）
- `limit`: 最大結果数（デフォルト: 20）

**スコア統合式**:
```
hybrid_score = local_score * local_weight + framework_score * (1 - local_weight)
```

**受け入れ基準**:
- [ ] ローカルとフレームワークの両方を並列検索
- [ ] 結果を統合スコアでランキング
- [ ] ソース（local/framework）を各結果に明示
- [ ] プロジェクト使用フレームワークを自動検出（package.json, requirements.txt等）
- [ ] ローカル結果とフレームワーク結果の比率を調整可能
- [ ] 結果のグルーピング（同一概念をまとめる）

**エラー処理**:
- フレームワーク未検出: ローカルのみの結果を返却（警告付き）

#### REQ-005: コード品質分析

**説明**: コードの品質メトリクスを分析（複雑度、結合度など）

**ユースケース**:
- `analyze_quality("OrderService")` → 循環的複雑度、結合度、凝集度を算出
- リファクタリング候補の自動検出

**メトリクス定義と閾値**:

| メトリクス | 説明 | 良好 | 警告 | 危険 |
|-----------|------|------|------|------|
| サイクロマティック複雑度 | 分岐の複雑さ | 1-10 | 11-20 | 21+ |
| 結合度（Afferent） | 被依存数 | 0-10 | 11-20 | 21+ |
| 結合度（Efferent） | 依存数 | 0-10 | 11-15 | 16+ |
| 凝集度（LCOM） | メソッド間の関連性 | 0-0.3 | 0.3-0.7 | 0.7+ |
| メソッド行数 | 関数の長さ | 1-30 | 31-50 | 51+ |
| パラメータ数 | 引数の数 | 0-4 | 5-7 | 8+ |

**受け入れ基準**:
- [ ] サイクロマティック複雑度の算出
- [ ] 結合度（Afferent/Efferent Coupling）の測定
- [ ] 凝集度（LCOM: Lack of Cohesion of Methods）の測定
- [ ] 閾値を超えた項目に警告フラグを付与
- [ ] リファクタリング提案（具体的なアクション）

#### REQ-006: コード進化追跡

**説明**: Git履歴と統合し、コードの進化を追跡

**ユースケース**:
- `track_evolution("AuthController")` → このクラスの変更履歴と関連コミットを表示
- ホットスポット（頻繁に変更されるコード）の検出

**パラメータ定義**:
- `entity_name`: 追跡対象エンティティ
- `since`: 開始日（デフォルト: 6ヶ月前）
- `until`: 終了日（デフォルト: 現在）
- `include_blame`: git blame情報を含むか（デフォルト: false）

**ホットスポットスコア算出**:
```
hotspot_score = change_frequency * 0.5 + unique_authors * 0.3 + lines_changed * 0.2
```

**受け入れ基準**:
- [ ] Git履歴との統合（GitPython使用）
- [ ] エンティティ単位の変更頻度分析
- [ ] ホットスポット検出（上位10%を警告）
- [ ] 変更者（author）の追跡
- [ ] コミットメッセージの関連付け
- [ ] 変更行数の集計
- [ ] 時系列グラフデータの出力（JSON形式）

**エラー処理**:
- Gitリポジトリ外: 「Gitリポジトリが見つかりません」エラー
- エンティティ未発見: 「エンティティが見つかりません」エラー

#### REQ-007: AIコード生成ガイダンス

**説明**: プロジェクトのコーディング規約に基づいたコード生成ガイダンス

**ユースケース**:
- `get_coding_guidance("create new service class")` 
  → プロジェクト内の既存パターンに基づいた雛形を提案

**パラメータ定義**:
- `task`: 生成タスクの説明
- `entity_type`: 生成するエンティティタイプ（class, function, module等）
- `similar_to`: 参考にするエンティティ名（オプション）

**出力内容**:
```json
{
  "naming_convention": "PascalCase for classes, snake_case for functions",
  "suggested_name": "UserAuthenticationService",
  "template": "class {name}:\n    def __init__(self):\n        ...",
  "similar_entities": [...],
  "import_suggestions": [...],
  "directory_suggestion": "src/services/"
}
```

**受け入れ基準**:
- [ ] プロジェクト内のパターン学習（命名、構造、インポート）
- [ ] 命名規則の推論（snake_case, camelCase, PascalCase等）
- [ ] 構造パターンの提案（類似エンティティから抽出）
- [ ] 適切なディレクトリ配置の提案
- [ ] 必要なインポート文の提案
- [ ] テンプレートコードの生成
- [ ] 類似エンティティ3件以上から学習

**エラー処理**:
- 類似エンティティ不足: 汎用テンプレートを提案（警告付き）

#### REQ-008: 設計パターン検出

**説明**: コードベースから設計パターンを自動検出

**ユースケース**:
- `detect_patterns()` → Singleton, Factory, Observer などのパターンを検出
- パターンの正しい使用/誤用を指摘

**対象パターン（Phase 1: 10パターン）**:

| カテゴリ | パターン | 検出ロジック |
|----------|----------|-------------|
| 生成 | Singleton | プライベートコンストラクタ + 静的インスタンス |
| 生成 | Factory Method | 抽象生成メソッド + 具象クラス |
| 生成 | Builder | 段階的構築メソッドチェーン |
| 構造 | Adapter | インターフェース変換ラッパー |
| 構造 | Decorator | 同一インターフェース + 委譲 |
| 構造 | Facade | 複数サブシステムへの統一インターフェース |
| 振る舞い | Observer | Subject/Observer登録パターン |
| 振る舞い | Strategy | インターフェース + 実行時切り替え |
| 振る舞い | Command | execute()メソッド + アクションカプセル化 |
| 振る舞い | Template Method | 抽象クラス + フックメソッド |

**受け入れ基準**:
- [ ] 上記10パターンの検出（精度80%以上）
- [ ] パターンの適用箇所の特定（ファイル、行番号）
- [ ] 信頼度スコア（0.0〜1.0）の付与
- [ ] 改善提案（誤用の場合）

## 8. リスクと対策

| リスク | 影響 | 確率 | 対策 |
|--------|------|------|------|
| パフォーマンス劣化 | 大規模プロジェクトで遅延 | 高 | キャッシュ、インデックス最適化、遅延評価 |
| メモリ使用量増加 | 知識グラフのメモリ消費 | 中 | 遅延ロード、LRUキャッシュ、メモリマップ |
| Git統合の複雑さ | REQ-006の実装難易度 | 中 | GitPythonライブラリ活用、段階的実装 |
| パターン検出の精度 | 誤検出/未検出のリスク | 高 | 保守的な閾値、信頼度スコア、ユーザー確認 |
| 大規模グラフ探索 | インパクト分析のタイムアウト | 中 | 深度制限、早期打ち切り、非同期処理 |
| 言語間の差異 | パーサー依存の一貫性欠如 | 低 | 言語別テストスイート、抽象化レイヤー |

## 付録A: Context7 vs YATA ユースケース比較

### Context7のユースケース
```
User: Create a Next.js middleware that checks for JWT. use context7
Context7: [外部ドキュメントを取得] → 最新のNext.jsミドルウェア例を返す
```

### YATA（強化後）のユースケース
```
User: Create a Next.js middleware that checks for JWT. use yata
YATA: 
1. [フレームワーク検索] Next.jsの認証パターンを取得
2. [ローカル検索] プロジェクト内の既存ミドルウェアを検索
3. [統合推薦] フレームワークのベストプラクティス + プロジェクトの規約に合わせたコードを提案
4. [インパクト分析] 新しいミドルウェアを追加した場合の影響を提示
```

## 付録B: 用語集

| 用語 | 定義 |
|------|------|
| エンティティ | コード内の識別可能な要素（クラス、関数、変数など） |
| リレーションシップ | エンティティ間の関係（calls, imports, inheritsなど） |
| 知識グラフ | エンティティとリレーションシップを格納するグラフ構造 |
| セマンティック検索 | 意味に基づいた検索（単純な文字列マッチングを超える） |
| インパクト分析 | 変更が他のコードに与える影響の分析 |
